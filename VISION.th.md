# วิสัยทัศน์ (Vision)

> ฉบับภาษาไทย — ต้นฉบับ (canonical) คือ [`VISION.md`](VISION.md) เมื่อเกิดข้อขัดแย้งระหว่างสองภาษา ฉบับภาษาอังกฤษเป็นเกณฑ์ตัดสิน

## วัตถุประสงค์

BWOC มีอยู่เพื่อให้ใครก็สร้าง AI coding agent ได้ บนฐานราก **มีหลักการและเป็นกลางต่อ backend** — ฐานรากที่ถือว่าความไม่เที่ยงของสถานะ (state impermanence), การสืบสาวความผิดพลาด, วงรอบชีวิต (lifecycle), ความเชื่อใจระหว่าง agent, และการประเมินภัยคุกคาม เป็นเรื่องสำคัญลำดับต้น ไม่ใช่สิ่งที่คิดทีหลัง

## สิ่งที่ BWOC จำลอง

BWOC ไม่ได้จำลอง "การเขียน code" แต่จำลอง **วงรอบทั้งหมดของ coding agent** ในฐานะสังขตธรรม — สิ่งที่เกิดขึ้น ตั้งอยู่ และดับไป

วงรอบนี้มีสามระยะ ตามไตรลักษณ์ที่ปรากฏใน **อังคุตตรนิกาย 3.47 (สังขตสูตร)** —

| ระยะ | บาลี | สิ่งที่ agent ทำ |
|---|---|---|
| เกิดขึ้น | **อุปฺปาท** | สร้างตัวตน — agent ถูก incarnate จาก template, กำหนด persona, ประกาศความสามารถ, resolve manifest |
| ตั้งอยู่ (พร้อมแปรไป) | **ฐิติ** | ดำเนินงาน — วางแผนด้วยอริยสัจ 4, ลงมือด้วยมรรค 8, จดจำด้วยสัมมาสติ, สื่อสารด้วยพรหมวิหาร 4 ภายในขอบเขตของ task หรือ session |
| ดับไป | **วยะ** | จบการทำ — ล้าง worktree, ปล่อย branch, ตัดความจำ, ปิด task ด้วยการบันทึก อนัตตา — ไม่ยึดมั่น |

วงรอบนี้คือรูปทรงเชิงสถาปัตยกรรมของทุกองค์ประกอบใน BWOC ตั้งแต่ task หนึ่ง ๆ session หนึ่ง ๆ ไปจนถึงชีวิตของ agent ทั้งตัว หลักธรรม 22 ประการใน [`PHILOSOPHY.th.md`](modules/agent-template/docs/th/PHILOSOPHY.th.md) ทั้งหมดอยู่ภายในระยะใดระยะหนึ่งของวงรอบนี้

## ช่องว่าง

กรอบวิศวกรรมตะวันตก (DDD, Clean Architecture, SOLID, Hexagonal) ลึกซึ้งในเรื่องโครงสร้างและการพึ่งพา แต่บางในเรื่องเหล่านี้ —

- **ระบบ *คือ* อะไรเมื่อเวลาผ่านไป** ในเมื่อสถานะเปลี่ยนตลอด
- **ทำไม** ความล้มเหลวจึงเกิด สาวกลับไปหาเหตุปัจจัย
- **อย่างไร** agent อิสระต่าง ๆ จึงเชื่อใจและร่วมงานกันโดยไม่มีศูนย์กลาง
- **อะไร** ที่ agent ควรปฏิเสธโดยอัตโนมัติ โดยไม่มีคนเขียนกฎอยู่ตรงนั้น

ระบบ agent ล้มเหลวในมิติเหล่านี้พอดี BWOC จึงนำกรอบทางพุทธมาใช้ เพราะกรอบเหล่านี้แม่นยำเป็นพิเศษเรื่องอนิจจัง, ปฏิจจสมุปบาท, เจตนา, และวินัย — เป็นเรื่องเดียวกัน แต่ใช้ศัพท์ต่างกัน

## แนวทาง

- **ศัพท์บาลีเป็นชื่อหัวข้อ เนื้อหาเป็นเชิงเทคนิค** ไม่ใช่การตีความศาสนา ผู้อ่านที่ไม่เคยรู้จักพุทธศาสนามาก่อนสามารถอ่านเอกสารและเขียน code ได้
- **หนึ่งสเปก หลาย backend** `AGENTS.md` คือแหล่งความจริงเพียงหนึ่งเดียว Claude, Antigravity, Codex, Kimi และ LLM ใด ๆ ในอนาคต อ่านไฟล์เดียวกันผ่าน symlinks ไม่มี backend ได้รับความสนใจเป็นพิเศษ
- **หนึ่ง repository หนึ่ง agent** แต่ละ agent อยู่ในที่อยู่ของตนเอง สามารถ fork ได้ ถูก clone มาจาก template ไม่มี runtime กลาง
- **เอกสารก่อน implementation** กรอบนี้ถูกสเปกครบใน Markdown ก่อนเขียน runtime code ใด ๆ code ตามหลักธรรม ไม่ใช่ตรงข้าม

## ความสำเร็จในหนึ่งปี

- ผู้ contribute สาธารณะ สามารถ clone template, กรอก `config.manifest.json`, แล้วได้ agent profile ที่ใช้งานได้ภายในไม่ถึง 30 นาที
- มี reference agent อย่างน้อยสามตัวอยู่ในธรรมชาติ สร้างโดยผู้ดูแลนอกทีมผู้เขียนต้นฉบับ
- หลักธรรม 22 ประการได้รับการ stress-test กับเหตุการณ์จริงใน agent ไม่ใช่แค่ออกแบบบน whiteboard
- มี translation อย่างน้อยสามภาษาเพิ่มเติมจากอังกฤษ ใช้โครงสร้าง `docs/<lang>/` ที่เป็นกลางต่อภาษา

## ความสำเร็จในสามปี

- ศัพท์ BWOC (Yoniso manasikāra checks, Mattaññutā caps, Sīla baselines, Kalyāṇamitta trust scores) ปรากฏใน codebase ของ agent ที่ไม่มีความสัมพันธ์อย่างเป็นทางการกับ project นี้
- รูปแบบ fleet governance ข้าม vendor (Aparihāniya-dhamma 7) อยู่ใน production ขององค์กรมากกว่าหนึ่งแห่ง
- กรอบนี้ผ่าน refactor ใหญ่ครั้งแรกของตนเองได้ โดย doctrine layer ไม่แตก

## Agent แบบ Self-Hosted และเป็นกลางต่อ Provider

BWOC ถือว่า **LLM แบบ self-hosted เป็นเป้าหมาย deployment ระดับแรก** ไม่ใช่สิ่งที่คิดทีหลัง crate `bwoc-harness` เพิ่ม OpenAI-compatible model-API client และ agentic loop runtime ทำให้ agent สามารถรันบน infrastructure ท้องถิ่นหรือส่วนตัวได้ทั้งหมด (Ollama, vLLM, LM Studio, llama.cpp server หรือ endpoint OpenAI-compatible ใดก็ตาม)

### การแบ่ง Orchestrator กับ Runtime

path `bwoc` เริ่มต้นยังคงเป็น **zero-dep CLI orchestrator**: exec vendor agentic CLI (Claude, Antigravity, Codex, Kimi) และไม่ต้องเรียก model-API ผู้ใช้ที่ทำงานกับ cloud backend เท่านั้นไม่ต้อง pull runtime

path `bwoc-harness` แบบ optional เพิ่ม **model-API client และ agent runtime** ที่มี safety production-grade (guardrails, sandbox, permission system), telemetry, และ offline eval framework dep หนัก (tokio, reqwest, keyring) ถูก **กักตัวไว้ใน `crates/bwoc-harness`** และไม่กระทบ crate graph เริ่มต้น การเพิ่ม self-hosted backend ใช้คำสั่งเดียว: `ln -s AGENTS.md OLLAMA.md`

นี่คือการขยายตัวตนอย่างตั้งใจจาก "pure orchestrator" เป็น "orchestrator + optional runtime" เพราะ self-hosted operation ต้องการการควบคุม safety, telemetry, และ evaluation ที่ third-party agentic CLI ไม่สามารถมอบให้ได้ สเปกที่เล็กกว่ายังคงชนะในทุกที่ที่การขยายตัวนั้นไม่จำเป็น

## สิ่งที่ไม่ใช่เป้าหมาย

- **ไม่ใช่ศาสนา** ไม่ใช่คู่มือทำสมาธิ ไม่ใช่ยานพาหนะของครู, สายธรรม, หรือสำนักใด ๆ
- **ไม่ใช่ตัวแทนของ DDD, Clean Architecture, หรือ SOLID** BWOC ขยายกรอบเหล่านั้นไปสู่มิติที่กรอบเหล่านั้นไม่ได้ออกแบบมาเพื่อจัดการ
- **ไม่ใช่ runtime ที่ผูกติด** path `bwoc` เริ่มต้นยังเป็น CLI orchestrator ที่ไม่มี model-API dependency runtime `bwoc-harness` เป็น opt-in ไม่ใช่บังคับ
- **ไม่เอนเอียง vendor** ไม่มี backend, hosting provider, vector store, หรือ tool ใด ได้รับการปฏิบัติเป็นพิเศษในเอกสารหลัก ความเท่าเทียม self-hosted กับ cloud backend คือเป้าหมายการออกแบบ ไม่ใช่สิ่งที่คิดทีหลัง
- **ไม่ใช่กรอบเพิ่มผลผลิต** BWOC เพิ่มประสิทธิภาพให้กับ agent ที่ *มีหลักการ, ตรวจสอบได้, กู้คืนได้* ไม่ใช่ time-to-first-token ที่เร็วที่สุด

## หลักการที่ตัดสินเมื่อสิ่งดีขัดกัน

เมื่อสองสิ่งดีขัดกัน หลักเหล่านี้เป็นตัวตัดสิน —

1. **สมานัตตตา** — ปฏิบัติกับทุก backend อย่างเท่าเทียม ชนะความสะดวกเฉพาะ vendor ใด ๆ
2. **มัตตัญญุตา** — สเปกเล็กกว่าชนะสเปกครบกว่า เว้นแต่ความครบถ้วนเป็นสิ่งจำเป็น
3. **โยนิโสมนสิการ** — สิ่งที่ตรวจสอบกับสถานะปัจจุบัน ชนะสิ่งที่จำมาจากอดีต
4. **ศีล** — เกณฑ์ความปลอดภัยส่วนกลาง ชนะความยืดหยุ่นของรายบุคคล
5. **อนัตตา** — การปล่อยสถานะเก่า ชนะการรักษาไว้ "เผื่อ"

เมื่อยังลังเล เลือกตัวเลือกที่ผู้ contribute สาธารณะที่ไม่มี context มาก่อน สามารถนำไปใช้ได้โดยไม่ต้องขออนุญาต
