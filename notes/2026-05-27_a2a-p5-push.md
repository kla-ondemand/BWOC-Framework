# 2026-05-27 — A2A P5: push-notification config management

The final #48 slice. Adds the four A2A push-notification **config** methods —
`CreateTaskPushNotificationConfig` + `Get`/`List`/`Delete` — and advertises
`capabilities.pushNotifications: true`. Completes A2A v1 (after P1-core #71,
P1-serve #72, P2 #73/#74, P3 #75, P4 #76).

## What changed

- **`bwoc-a2a::push`** (new) — `PushConfig { taskId, id, url, token? }`, plus
  `load`/`save`/`configs_path` for a team-scoped `push-configs.json` (sits next
  to `tasks.jsonl`).
- **`rpc.rs`** — four handlers: Create (validates the task exists, assigns a
  `pnc-<nanos>` id, persists), Get (by config id), List (per task → `{configs:[…]}`),
  Delete (by id). Accept `pushNotificationConfigId`/`configId`/`id` aliases.
  No team exposed → `-32001` (configs are task-scoped, tasks need a team).
- **`card.rs`** — `pushNotifications: true`.

## Decisions

- **CRUD only — webhook delivery is deferred to the auth phase.** Actually
  POSTing task updates to a registered webhook is a network egress that, under
  P1's no-auth posture, is **both an SSRF amplifier** (webhook → internal
  services / cloud metadata) **and a data-exfil path** (a client registers an
  external sink for a task's updates it shouldn't receive). Shipping that on an
  unauthenticated listener contradicts the conservative loopback/defer-auth
  stance held across the epic. So P5 manages configs (real, persisted, testable)
  and defers delivery to the auth phase, where the webhook URL can be constrained
  and the registrant authenticated. The inert-config gap is documented loudly in
  the module rather than implying notifications fire (Musāvāda). *(Operator-chosen
  among: full-push-with-SSRF-guard / CRUD-only / stub.)*
- **Team-scoped persistent store.** Configs live in
  `.bwoc/teams/<team>/push-configs.json` — task-scoped, beside the task list they
  reference, surviving restarts (unlike an in-memory store).
- **`pushNotifications: true` reflects "the config API works".** A peer that
  registers a config can read it back; the delivery gap is the documented
  limitation. (Advertising the config-management capability, not a delivery SLA.)
- **The registrant `token` is stored but never echoed.** Get/List responses omit
  it — under the no-auth listener any local caller could `List` and read tokens
  another caller registered. It's persisted (for the auth-phase delivery path),
  not returned over the wire.

## Alternatives considered

- *Full push + SSRF guard now.* Rejected for P1 — even with a private-IP block,
  the no-auth listener still lets any local client exfil a task's updates to an
  external sink. Delivery belongs behind auth.
- *Stub (`pushNotifications:false` + `-32003`).* Rejected — the config CRUD is
  genuinely useful and safe to ship; stubbing it would under-deliver the phase.
- *In-memory config store.* Rejected — configs should survive a `serve` restart,
  and BWOC's model is file-based.

## Status / deferred

- Done: `push` module + 4 RPC handlers, `pushNotifications:true`, 45 `bwoc-a2a`
  tests (push store round-trip; HTTP-layer Create→List→Get→Delete→gone; unknown-task
  + no-team → `-32001`).
- Hardened from review: the config id is `pnc-<nanos>-<per-process counter>`
  (collision-free even within a clock tick); `save` writes a sibling `.tmp` then
  renames (no crash-truncation) and chmods the file to `0600` on Unix (it can
  hold a bearer token — matches `.bwoc/agent.key`); both `Get` and `Delete` are
  keyed by `(taskId, configId)` so a caller can't read or remove a config under a
  task it didn't name.
- Deferred (auth phase): **webhook delivery** (the background watcher that POSTs
  task-status updates), SSRF guard + registrant auth, `AuthenticationInfo` beyond
  a bare token, token **encryption**-at-rest (only `0600` perms today), a per-task
  config cap, and a cross-process advisory lock (atomic *rename* is in place,
  *locking* is not — low-frequency, loopback, single-agent for now).
- This closes the #48 A2A interop epic at v1: discovery + inbound `SendMessage`,
  tasks bridge, SSE streaming, outbound client, and push-config management — all
  with HTTP deps quarantined to the `bwoc-a2a` crate (`bwoc-cli` stays HTTP-free).
