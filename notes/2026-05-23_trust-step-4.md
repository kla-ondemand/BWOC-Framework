# 2026-05-23 — Trust Step 4: Daemon-Side Refusal

Step 4 of the 5-step Kalyāṇamitta-7 trust implementation order from
[`interconnect/trust.md`](../modules/agent-template/interconnect/trust.md#implementation-order-when-code-work-begins).
The `bwoc-agent --serve` daemon now refuses inbox envelopes from senders
who lack required trust qualities, gated behind a `BWOC_TRUST_GATING=1`
env opt-in (v1 safety). Refusals are auditable: original envelopes are
never deleted; a sidecar JSONL log records the verdict, and `bwoc inbox`
merges the refusal verdict into the envelope's JSON so the spec's
`jq '.[] | select(.refused)'` example works end-to-end.

## What changed

- **`bwoc-core::time` (new module)** — promoted `utc_now_iso8601` +
  `format_iso8601` out of `bwoc-cli::util` so `bwoc-agent` can use the
  same proleptic-Gregorian formatter without duplicating it. The util
  module's pre-existing TODO comment ("Promote to `bwoc-core` if a second
  crate ever needs them") said exactly when to do this; step 4 was when.
  `bwoc-cli::util` now re-exports from core for backward compat with
  every existing caller (`send`/`supervise`/`init`/`new`).
- **`TrustBlock::missing_in()` + `TrustDeclared::has()` helpers** — pure
  functions in `bwoc-core::manifest`. `has(key)` reads a quality by its
  camelCase manifest key with the spec's "missing → false" rule for
  unknown keys (forward-compat seam for v2 qualities like `mudu`).
  `missing_in(declared)` returns the qualities the recipient's
  `required_trust` demands that the peer's `declared` lacks. 7 new unit
  tests covering: known-keys, unknown-keys, empty-required, all-satisfied,
  partial-missing, order-preservation, future-quality-always-missing.
- **`bwoc-agent::trust` (new module)** — daemon-side refusal evaluation.
  `TrustContext::build(own_manifest, cwd)` snapshots the recipient's
  `requiredTrust`, walks ancestors for `.bwoc/workspace.toml`, and reads
  `BWOC_TRUST_GATING` at startup. `evaluate(ctx, line, offset)` short-
  circuits when inert (gating off OR required empty), special-cases
  `from=user` (always passes per spec), and otherwise resolves the
  sender's manifest via the workspace registry to compute the missing
  list. Returns a `Refusal { envelope_offset, envelope_ts, envelope_from,
  reason, missing }` ready to serialize. 8 new unit tests.
- **Daemon poll loop wires the trust context in** — `serve_loop` now
  takes the recipient's `Manifest`, builds the `TrustContext` once at
  startup (logs gating state to stderr), and threads it + the
  `inbox.refusals.jsonl` path into `check_inbox_for_new`. On each new
  envelope: if refusal verdict fires, append a JSONL record to the
  sidecar log and announce as `inbox REFUSED ← from: reason=… missing=…`
  on stderr instead of the normal `inbox ←` announcement.
- **`bwoc inbox` merges refusals into the envelope view** — `read_messages`
  now reads `inbox.refusals.jsonl` alongside `inbox.jsonl`, joining by
  byte offset (the daemon records `envelopeOffset` per refusal; the
  reader tracks byte offsets per inbox line). When a match exists, the
  envelope's JSON gets a `refused: { reason, missing }` field. 4 new
  unit tests covering: byte-offset tracking, refusal-merge, missing
  sidecar = silent passthrough, malformed refusal-line skipped.

## Decisions

- **Sidecar log over in-place envelope rewrite.** The spec literally
  says "writes a refused field on the envelope," but modifying
  `inbox.jsonl` in place breaks the daemon's byte-cursor invariant and
  races against `bwoc send` appends. The sidecar approach preserves
  append-only semantics on both files, eliminates the race entirely,
  and the spec's user-facing example (`jq '.[] | select(.refused)'`)
  still works because `bwoc inbox` merges the two files at read time —
  consumers see the same shape the spec promises.
- **Byte offset as the join key**, not (ts, from, message). Byte offset
  is a stable identity within a single inbox session — `bwoc send`
  always appends, so once a line is written it stays at that position
  until a `--clear` truncates the whole file (which also drops the
  sidecar's relevance for those offsets). Three-tuple matching would
  collide on rapid-fire duplicate sends.
- **Gating env opt-in (BWOC_TRUST_GATING=1) is per the spec's safety
  recommendation**, not invented here. The spec says step 4 "should
  ship behind a `BWOC_TRUST_GATING=1` env opt-in initially" precisely
  because trust evaluation has runtime risk (a wrong refusal blocks
  real messages). Once stable in v2, the env disappears and behavior
  is controlled purely by recipient's `requiredTrust`.
- **`from=user` always passes.** Spec section "Refusal Semantics" is
  explicit: "Sender == `user` is a special case: user-originated
  messages always pass (the user is by definition above the trust
  gate)." Implemented as the first check after gating-on.
- **`no_workspace` / `unknown_sender` / `sender_manifest_unreadable`
  all refuse with the full required list as `missing`.** When we can't
  verify, we can't certify — the user opted in to gating, so the
  stricter interpretation is correct. Reason field is distinct from
  `missing_trust` so the operator can distinguish "real failure" from
  "couldn't tell."
- **TrustContext built once at startup, registry re-read per envelope.**
  Daemons run for hours; new agents incarnated during the daemon's
  lifetime (via `bwoc new` from another shell) need to be reachable
  by the trust lookup. Inbox traffic is rare, registry I/O is cheap,
  so per-envelope load is fine.
- **TrustContext's `is_inert()` short-circuits before JSON parsing.**
  When gating is off OR `required` is empty, no envelope ever gets
  parsed for trust purposes — same hot path as before this change.
  Mattaññutā: pay for trust only when you opted in to it.

## Alternatives considered

- **In-place envelope rewrite.** Read inbox.jsonl, mutate the refused
  lines to add `refused: {...}`, write back atomically via tmp +
  rename. Rejected because: (a) breaks the daemon's byte-cursor (line
  lengths change), (b) races against `bwoc send` appends, (c) requires
  an advisory lock to be correct, which would add a dep. Sidecar +
  merge-at-read achieves the same `select(.refused)` surface without
  any of those costs.
- **`inbox.refusals.jsonl` line-by-line mirrors `inbox.jsonl`.** Empty
  line N = no refusal for envelope N; non-empty = refusal. Rejected
  because it requires lock-step between sends (which happen at random
  times) and refusals (which happen at the daemon's poll cadence) —
  the very race we were trying to avoid. Offset-keyed sidecar is the
  same data shape without the synchronization burden.
- **Refusal record uses (envelopeTs, envelopeFrom) as join key.**
  Considered for collision-safety paranoia, but byte offset is more
  robust: two `bwoc send` calls with the same `--message` in the same
  second from the same operator (the only way ts+from collides in
  practice) still have distinct offsets.
- **Refuse unknown senders silently (log only, don't write to
  sidecar).** Rejected — the operator who turned on `BWOC_TRUST_GATING=1`
  wants to see refusals. Silent refusal is the worst kind of policy
  surface: invisible behavior that the user must reverse-engineer.

- bwoc-agent: 15 tests (+ 8 new for `trust::evaluate` + `TrustContext`)
- bwoc-cli: 65 tests (+ 4 new for envelope-offset reading + refusal merge)

Total 99 passing, 0 failures. `cargo clippy --workspace --all-targets`
clean. Live verification: spawned a sender + recipient in a scratch
workspace, set recipient's `requiredTrust` to `["vatta", "noCatthana"]`,
appended an envelope from `agent-sender` (no trust block — guaranteed
to fail), and confirmed:
1. Daemon stderr logged `inbox REFUSED ← agent-sender: reason=missing_trust missing=["vatta", "noCatthana"]`.
2. `from=user` envelopes still log as `inbox ← user:` (pass-through).
3. `inbox.refusals.jsonl` has the JSON record with `envelopeOffset=0`.
4. `bwoc inbox recipient --json` attaches `refused: {reason, missing}` only to the refused envelope.
5. `bwoc inbox recipient --json | jq -c '.messages[] | select(.refused)'` emits just the refused envelope — spec example works verbatim.
6. With `BWOC_TRUST_GATING` unset: same envelope passes through, no
   sidecar file created.
