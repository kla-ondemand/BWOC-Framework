## 0. ขอบเขต และชื่อ "Trust v2"

> [!warning] มีสองชั้นที่ใช้คำว่า "trust" — ต้องแยกให้ชัด
> `interconnect/trust.md` มีฟีเจอร์ชื่อ **"Trust v2" (warn-mode, v2026.5.24)** อยู่แล้ว: โมเดล **Kalyāṇamitta-7** — บูลีน 7 ตัวที่ตัวแทนประกาศเอง ใช้ให้ผู้รับตัดสิน *ว่าจะรับข้อความจากผู้ส่งหรือไม่ตั้งแต่แรก* นั่นคือ **การให้สิทธิ์ (authorization)**: *ฉันควรฟังเพื่อนรายนี้ไหม?*
>
> เอกสารนี้ระบุชั้นที่ **ต่ำกว่าและต่างออกไป**: **การพิสูจน์ตัวตน (authentication)** — *ข้อความนี้มาจากตัวแทนที่อ้างจริงหรือไม่?* การตัดสินด้วย Kalyāṇamitta-7 จะเชื่อถือได้ก็ต่อเมื่อ "ตัวตนของผู้ส่งเป็นของจริง" ซึ่งวันนี้เป็นเพียงการ *อ้าง* ไม่ใช่การ *พิสูจน์*

| ชั้น | คำถาม | กลไก | ที่อยู่ |
|---|---|---|---|
| Authorization (ปล่อยแล้ว) | ฉันรับข้อความจากเพื่อน *ประเภทนี้* ไหม? | บูลีน Kalyāṇamitta-7 + `requiredTrust` | `interconnect/trust.md` |
| **Authentication (สเปกนี้)** | ข้อความมาจากเพื่อนรายนั้นจริงไหม? | **ซองข้อความเซ็นด้วย ed25519** | inbox + guardrails ของ harness |

เนื่องจากชื่อชนกัน **§9 จึงขอให้ผู้ดูแลรับรองชื่อไฟล์** (คง `TRUST-V2` พร้อมการแบ่งชั้นนี้ หรือเปลี่ยนเป็นเช่น `SIGNING.en.md` / `MESSAGE-AUTH.en.md`)

## 1. ภัยคุกคามที่จัดการ

> ตัวแทนปลอมค่า `sender` ของข้อความใน inbox เพื่อสวมรอยเป็นเพื่อนที่ได้รับความไว้วางใจ และเลี่ยงกฎปฏิเสธ Kalyāṇamitta-7 ของผู้รับ

ปัจจุบันตัวตนถูกตรวจด้วย `check_identity_spoof()` (`crates/bwoc-harness/src/policy/guardrails.rs:293` เรียกที่ `:81`) — เป็น **ฮิวริสติกแบบจับคำ/สตริงย่อย** จับได้แค่สตริงหยาบ ๆ อย่าง `"I am agent-X"` แต่ **พิสูจน์การเป็นผู้เขียนไม่ได้** และเลี่ยงได้ง่ายด้วยการเปลี่ยนถ้อยคำ ไม่มีการผูกเชิงเข้ารหัสระหว่างข้อความกับผู้ส่งที่อ้าง

สิ่งนี้แมปกับ **Musāvāda** (มุสาวาท): ตัวตนปลอมคือการโกหกระดับระบบ ทางแก้คือทำให้ "การเป็นผู้เขียนที่แท้จริง" *ตรวจสอบได้* เพื่อให้การโกหกถูกตรวจจับได้ ไม่ใช่แค่ถูกห้ามปราม

## 2. ทำไมต้อง ed25519 ไม่ใช่ HMAC

MAC แบบกุญแจร่วม (HMAC) ล้มเหลวในกรณี **ข้าม workspace** ซึ่งเป็นแรงจูงใจของงานนี้ (#20, give-feedback ข้ามขอบเขตความไว้วางใจ):

- กุญแจร่วมต้องแจกให้ทุกฝ่ายที่ต้อง *ตรวจสอบ* และผู้ใดถือกุญแจตรวจสอบก็ *ปลอมลายเซ็น* ได้ด้วย เพราะกุญแจเดียวกันใช้ทั้งเซ็นและตรวจ
- เมื่อข้ามขอบเขตความไว้วางใจ ไม่มีช่องทางปลอดภัยให้แชร์ความลับเช่นนั้น และสำเนาที่รั่วเพียงชุดเดียวก็ปลอมได้แทนทุกคน

**ed25519** (อสมมาตร) แยกสองบทบาทออกจากกัน: กุญแจ **ส่วนตัว** ใช้เซ็นเท่านั้น กุญแจ **สาธารณะ** ใช้ตรวจสอบเท่านั้น การเผยแพร่กุญแจสาธารณะทำให้ใครก็ตรวจสอบได้แต่ *ไม่มีใคร* ปลอมได้ ed25519 มีขนาดเล็ก (กุญแจ 32 ไบต์ ลายเซ็น 64 ไบต์) เร็ว และไม่มีพารามิเตอร์ให้พลาด

> [!note] หมายเหตุ dependency (dep-quarantine) — **รับรองแล้ว**
> ed25519 (`ed25519-dalek` v2 + `rand_core` + `hex`) อยู่ใน crate แยกแบบ lean ชื่อ **`bwoc-signing`** — *ไม่ใช่* `bwoc-core` (dep-quarantine ห้าม crypto) และ *ไม่ใช่* `bwoc-harness` (จุดเซ็น `bwoc send` กับจุด verify `bwoc-agent` ไม่ได้ depend harness) ทั้ง `bwoc-cli` และ `bwoc-agent` depend `bwoc-signing`; crate นี้ไม่ลาก async/HTTP จึงให้ `bwoc-core` คง lean ได้

## 4. ซองข้อความที่เซ็นแล้ว

ทุกข้อความระหว่างตัวแทนถูกหุ้มด้วยซองที่เซ็นแล้ว ลายเซ็นครอบคลุม serialization แบบ canonical ของทุกฟิลด์ยกเว้น `sig`

```jsonc
{
  "sender":    "agent-luban",        // ผู้เขียนที่อ้าง (กุญแจสาธารณะที่ใช้ตรวจ)
  "recipient": "agent-erlang",       // ผูกไว้: ซองที่ดักได้จะส่งต่อให้ฝ่ายที่สามไม่ได้
  "nonce":     "b3f1…",              // ไม่ซ้ำต่อคู่ (sender, recipient); กันการเล่นซ้ำ
  "ts":        "2026-05-26T10:00:00Z", // เวลาออกซอง; กำหนดกรอบเวลาเล่นซ้ำ
  "payload":   { /* เนื้อหาข้อความ */ },
  "sig":       "ed25519(sender_priv, canonical({sender,recipient,nonce,ts,payload}))"
}
```

- **`recipient` ถูกเซ็น** ดังนั้นซองที่ถูกต้องซึ่งถูกดักระหว่างทางจะถูกเปลี่ยนเป้าหมายไปยังตัวแทนอื่นไม่ได้
- **`nonce` + `ts`** รวมกันป้องกันการเล่นซ้ำ (ดู §5)
- Serialization แบบ canonical (ลำดับฟิลด์, การเข้ารหัส) ต้องถูกกำหนดตายตัวและเหมือนกันทั้งสองฝั่ง **(§9: เลือกรูปแบบ canonical — JSON เรียงคีย์ หรือ binary concat แบบ length-prefixed)**

## 5. การตรวจสอบ + กรอบเวลาเล่นซ้ำ

เมื่อรับซอง ก่อนการตรวจให้สิทธิ์ Kalyāṇamitta-7 ผู้รับจะ:

1. หา​กุญแจสาธารณะของ `sender` (ผู้ส่งไม่รู้จัก → ปฏิเสธ)
2. ตรวจ `sig` เหนือ `canonical({sender,recipient,nonce,ts,payload})` (ล้มเหลว → ปฏิเสธ)
3. ตรวจว่า `recipient` คือตัวแทน *นี้* (ไม่ตรง → ปฏิเสธ)
4. ตรวจว่า `ts` อยู่ในช่วง `±skew` จากเวลาปัจจุบัน (ค่าเริ่มต้น **±5 นาที**, §9) (เก่า/อนาคต → ปฏิเสธ)
5. ตรวจว่า `nonce` ยังไม่เคยพบในกรอบเลื่อนของ `sender` นี้ (เคยพบ → ปฏิเสธว่าเป็นการเล่นซ้ำ)
6. บันทึก nonce แล้วจึงส่ง payload ต่อไปยังชั้น Kalyāṇamitta-7

**กรอบ nonce แบบเลื่อน:** ผู้รับเก็บคู่ `(sender, nonce)` ที่เพิ่งพบพร้อม `ts` แล้วทิ้งรายการที่เก่ากว่าขอบเขต skew หน่วยความจำถูกจำกัดด้วย `skew × อัตราข้อความ` กลไกนี้ *แทนที่* ฮิวริสติกจับคำ `check_identity_spoof()`

> [!note] บันทึก audit เป็นสิ่งที่ **สร้างใหม่** ไม่ใช่ seam ที่มีอยู่
> โน้ตวางแผนอ้าง `inbox.refusals.jsonl` ว่าเป็น "seam" — แต่วันนี้ **ไม่มี** (ไม่พบ `refusals` ที่ใดใน harness) สเปกนี้ **เสนอ** ให้ทำบันทึกการปฏิเสธ/audit แบบ append-only (เช่น `inbox.refusals.jsonl`) เก็บแต่ละซองที่ถูกปฏิเสธ `{ts, sender, recipient, reason}` เพื่อความสังเกตได้ ถือเป็นส่วนหนึ่งของ workstream นี้ ไม่ใช่ hook ที่มีอยู่ก่อน

## 6. การย้ายจาก `check_identity_spoof`

- การตรวจสอบ (§5 ข้อ 1–5) แทนที่ `check_identity_spoof()` (`guardrails.rs:293`)
- **ทยอยจาก warn → enforce** เลียนแบบวิธีที่ Kalyāṇamitta-7 ปล่อย warn-mode ก่อน: ข้อความที่ไม่ได้เซ็นหรือตรวจสอบไม่ผ่านจะ **เตือน** (และบันทึก) ในช่วงแรก แล้วจึง **ปฏิเสธ** เมื่อตัวแทนเผยแพร่กุญแจแล้ว ใช้ enum สไตล์ `vetted_mode` (`off` / `warn` / `enforce`) เพื่อคงความเข้ากันได้ย้อนหลังระหว่างการเปลี่ยนผ่าน
- ข้อความจากตัวแทนที่ยังไม่เผยแพร่กุญแจ: เป็นไปตามโหมดการทยอย (warn-mode รับพร้อมบันทึก; enforce ปฏิเสธ)

## 8. นอกขอบเขต (ของสเปกนี้)

- การ **เข้ารหัส** payload (ความลับ) สเปกนี้เป็นเรื่อง authentication/integrity เท่านั้น ข้อความถูกเซ็น ไม่ถูกเข้ารหัส
- PKI / certificate authority กุญแจเผยแพร่พร้อมตัวตนของตัวแทน (เชื่อตอนเผยแพร่) ไม่ผูกโซ่ไปยัง root
- UX การหมุนกุญแจและ revocation list (ร่างไว้ใน §3; รายละเอียดเลื่อนออกไป — §9)

## 9. การตัดสินใจที่รอผู้ดูแลรับรอง

นี่คือเกต แต่ละข้อมีข้อเสนอแนะ; รับรอง (✓) หรือเปลี่ยน ก่อนเขียนโค้ด crate ใด ๆ:

1. **ชื่อไฟล์ / ชื่อเรียก** → **เสนอ: เปลี่ยนเป็น `SIGNING.{en,th}.md`** เอกสารนี้ว่าด้วย *authentication* ของข้อความ ส่วน "Trust v2" เป็นฟีเจอร์ *authorization* Kalyāṇamitta-7 ใน `interconnect/trust.md` อยู่แล้ว ชื่อ `SIGNING` ระบุกลไกได้ตรงและจบการชนกันถาวร (ทางเลือก: คง `TRUST-V2` พร้อมหมายเหตุการแบ่งชั้นใน §0)
2. **แหล่งหลักของกุญแจสาธารณะ** → **เสนอ: manifest ของตัวแทน (`config.manifest.json`) เป็นหลัก; `interconnect/trust.md` อ้างถึง** manifest เป็นบันทึกตัวตนแบบเครื่องอ่านได้ที่ `bwoc check` ตรวจอยู่แล้ว ส่วน descriptor ยังเป็นเอกสารฝั่งมนุษย์ แหล่งความจริงเดียวที่ถูกตรวจสอบ ไม่มี sync drift
3. **Crate เซ็น** → **เสนอ: `ed25519-dalek` v2** (ผ่านการ audit, pure-Rust, ไม่มี network) + `rand_core` สำหรับสร้างกุญแจ **ไม่** feature-gate — การเซ็นเป็นแกนของเส้นทางความไว้วางใจ ไม่ใช่ของเสริม dep-quarantine ยังคงอยู่: อยู่ใน `bwoc-harness` เท่านั้น (ไม่ใช่ `bwoc-core`) เช่นเดียวกับ `keyring`/`landlock`
4. **Serialization แบบ canonical** → **เสนอ: RFC 8785 (JCS) canonical JSON** เหนือฟิลด์ที่เซ็น — เรียงคีย์, UTF-8, ไม่มี whitespace ที่ไม่จำเป็น อ่าน-debug ได้ และไม่ผูกกับภาษา (เพื่อนข้าม workspace อาจไม่ใช่ Rust); เลี่ยงกับดักของรูปแบบ binary เฉพาะกิจ
5. **กรอบเล่นซ้ำ** → **เสนอ: skew ของ `ts` ±5 นาที; เก็บ `(sender, nonce)` ที่พบไว้ `2×skew` (10 นาที) แล้วทิ้ง** ปฏิเสธ `ts` นอก ±5 นาที และ nonce ที่ยังอยู่ในกรอบ ทนความคลาดเคลื่อนของนาฬิกาโดยพื้นที่เล่นซ้ำจำกัด; หน่วยความจำ ≈ skew × อัตราข้อความ
6. **การทยอยปล่อย** → **เสนอ: enum โหมด `off | warn | enforce`, ค่าเริ่มต้น `warn`** (เลียน `vetted_mode`) ปล่อย warn ก่อน แล้วสลับเป็น `enforce` เมื่อตัวแทนเผยแพร่กุญแจแล้ว เข้ากันได้ย้อนหลัง ตรงกับที่ Kalyāṇamitta-7 และ `vetted_mode` ปล่อยมา
7. **บันทึกการปฏิเสธ/audit** → **เสนอ: ใช่ — `inbox.refusals.jsonl` แบบ append-only**, schema `{ts, sender, recipient, reason, mode}`, หนึ่งอ็อบเจกต์ต่อบรรทัด **ไม่มี payload/ความลับ** ตรงกับ convention ของ `session-metrics.jsonl` และป้อนให้ retrospective ของ HV2-3
8. **การหมุน/เพิกถอนกุญแจ** → **เสนอ: เลื่อนเป็นงานต่อเนื่อง** HV2-4 ส่ง สร้าง → เผยแพร่ → เซ็น → ตรวจ → ทยอย warn/enforce ส่วนการหมุน (เผยแพร่ใหม่ + ปลดเก่า) และ revocation list เป็นดีไซน์ใหญ่กว่า ขอบเขต HV2-4 ไว้ที่กลไกแกน (Mattaññutā)
