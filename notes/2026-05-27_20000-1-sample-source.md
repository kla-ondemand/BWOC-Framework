# 20000-1 Runtime — Sample Source & Evidence-Kind Split (BWOC-33)

Captures the two implementation decisions for replacing `audit-iso-20000-1/audit.sh` stub with a runnable that emits both `attestation` and `sample` evidence kinds per the [BWOC-26 evidence model](2026-05-27_iso-runtime-evidence-model.md) and the [BWOC-27 schema](../docs/en/PLUGINS.en.md#evidence-kinds). 9001 (BWOC-28) needed only one kind (`attestation`); 20000-1 is the first runtime that mixes two, so two new decisions arise that 9001 did not face:

1. **Which criterion uses which evidence kind** — the attestation/sample split across the eight declared criteria.
2. **Where the `sample` data comes from** — the analogue of BWOC-28's "where do attestations come from" question, but for rate-over-window evidence.

The attestation half reuses BWOC-28's mechanism verbatim. No dispatcher change is needed: `crates/bwoc-cli/src/audit.rs` already validates both `attestation` and `sample` kinds (extended in BWOC-28). BWOC-33 is plugin-only plus an e2e test.

## Decision 1 — Evidence-kind split across the eight criteria

The eight `criterion_id`s are STABLE (carried over unchanged from the v0.1.0 stub — PLUGINS.en.md §Stability). The runtime only changes how each criterion's finding is computed. The split follows one rule:

> **Documented-artifact criteria → `attestation`. Operational-rate criteria → `sample`.**

A criterion whose conformance is "a thing is documented / established / maintained" reduces to an operator-signed assertion (attestation). A criterion whose conformance is "N of M items met the bar over a window" is statistical (sample). This is the same documented-vs-operational distinction the [BWOC-26 matrix](2026-05-27_iso-runtime-evidence-model.md#standard--evidence-kind-matrix) drew for 20000-1 ("sample (incidents, changes, SLAs) + attestation (policy)"), extended consistently to the four criteria that matrix left unenumerated.

| `criterion_id` | Clause | Kind | Why |
|---|---|---|---|
| `20000-1-service-management-system-scope` | 4.3 | **attestation** | Scope is a documented artifact the operator vouches for ("SMS scope covers services X, Y, Z"). No population to sample. |
| `20000-1-service-policy-and-objectives` | 5.2 | **attestation** | Service policy — called out explicitly in BWOC-26 and PLUGINS.en.md §Evidence kinds as the 20000-1 attestation clause. |
| `20000-1-service-catalogue` | 8.3.1 | **attestation** | A maintained catalogue is a documented artifact; "is it current and complete" is an attestation by the service owner. |
| `20000-1-service-level-management` | 8.3.3 | **sample** | SLA conformance is a rate over a window ("N of M SLAs met"). |
| `20000-1-change-management` | 8.5.1 | **sample** | Change-process conformance is a rate ("N of M changes followed record→assess→approve→review"). |
| `20000-1-incident-management` | 8.6.1 | **sample** | The canonical rate-based clause ("49 of 50 incidents resolved within SLA") — the worked example in BWOC-26 and PLUGINS.en.md. |
| `20000-1-problem-management` | 8.6.3 | **sample** | Parallel to incident management — rate of problems with documented root cause / workaround. |
| `20000-1-continual-improvement` | 10.2 | **sample** | Improvement throughput is a count over a window ("N of M raised improvements implemented"). |

**Result: 3 attestation + 5 sample.** This satisfies the BWOC-33 brief ("most criteria are sample-based; 5.2 service policy uses attestation") — five of eight is the majority, and 5.2 is attestation. The brief delegated the exact split and the sample-source choice to this note; 4.3 and 8.3.1 are attestation rather than sample because neither has a population to sample (forcing a synthetic N-of-M onto "is the scope documented" would falsify the evidence — Musāvāda).

The per-criterion kind is **declared in `criteria.toml`** via the BWOC-26 `expected_evidence_kind` field plus the BWOC-29 per-kind `required` subtable, not hard-coded in `audit.sh`. `criteria.toml` becomes the single source of truth for the split; `bwoc check` (BWOC-29) statically validates it; the runtime reads it at invoke time and routes to the matching workspace.toml array. Adding `expected_evidence_kind` is additive (no id rename, no reorder) — minor bump, stability intact.

## Decision 2 — Where does `sample` data come from?

Same three options BWOC-28 weighed for attestations:

| Option | Where | Verdict |
|---|---|---|
| **A. workspace.toml block** | `.bwoc/workspace.toml [[plugins.audit-iso-20000-1.samples]]` | **Chosen** |
| B. Separate `samples/20000-1.toml` | New file / convention | Rejected — new path to learn, drift risk vs the `enabled` toggle. |
| C. Live ticket-system query | `command`-kind evidence or integration plugin hitting Jira/ServiceNow | Deferred — real value, but out of scope for v0.2.0; needs an integration contract that does not exist yet. |

### Decision — **Option A: `[[plugins.audit-iso-20000-1.samples]]` array of tables**

Rationale is the same as BWOC-28's attestation choice, and the BWOC-33 brief explicitly asks for consistency with it:

- **Smallest blast radius.** Same file the operator already touches to enable the plugin and to declare attestations. One mechanism, two arrays (`attestations` + `samples`), under one `[plugins.audit-iso-20000-1]` namespace.
- **Diff-able audit trail.** Every recorded metric shows up in `git log` — itself a meta-record that the SMS is being measured over time.
- **Reproducible (Musāvāda guard).** PLUGINS.en.md §Fields requires evidence be reproducible: "an operator running the same check by hand finds the same artifact." A committed sample block satisfies this — the artifact is the recorded number in `workspace.toml`, not an ephemeral API response that differs every query.
- **Discoverable.** `bwoc check` / `bwoc plugin list` already walk `workspace.toml`.

The operator transcribes the measured rate from their ITSM tool (incident tracker, change board, SLA dashboard) into the committed block. The runtime does **not** query the ticket system — live integration (Option C) is the v3 enhancement the BWOC-26 note anticipated ("the artefact lives in a ticket system, not the workspace"). v0.2.0 makes the operator the bridge: they pull the quarter's numbers and record them, the same way they sign an attestation.

### Schema in workspace.toml

```toml
[plugins.audit-iso-20000-1]
enabled = true

# attestation criteria (4.3, 5.2, 8.3.1) — identical shape to audit-iso-9001
[[plugins.audit-iso-20000-1.attestations]]
criterion_id  = "20000-1-service-policy-and-objectives"
statement     = "Service management policy v2.1 ratified 2026-01-15; objectives reviewed quarterly."
signer        = "Service Owner: Tonkla K."
signed_at     = "2026-01-15"
valid_through = "2027-01-15"   # optional

# sample criteria (8.3.3, 8.5.1, 8.6.1, 8.6.3, 10.2)
[[plugins.audit-iso-20000-1.samples]]
criterion_id  = "20000-1-incident-management"
summary       = "49 of 50 incidents resolved within SLA"
sampled_count = 49
sampled_of    = 50
window        = "2026-Q1"   # optional
```

- `[[plugins.audit-iso-20000-1.attestations]]` — same fields as 9001: `criterion_id`, `statement`, `signer`, `signed_at`, optional `valid_through`.
- `[[plugins.audit-iso-20000-1.samples]]` — `criterion_id`, `summary` (→ `evidence.value`), `sampled_count`, `sampled_of` (integers), optional `window`. (`valid_through` on samples is deferred to a later version — v0.2.0 keeps the sample surface to the schema's named sample fields.)
- `criterion_id` MUST match an entry in `criteria.toml`. First occurrence wins (duplicates are a `bwoc check` concern, not the runtime's).

## Decision 3 — Pass/fail semantics for `sample` criteria

When does a sample criterion pass? Two models:

| Model | Verdict |
|---|---|
| **Threshold-based** — `sampled_count / sampled_of ≥ X%` → pass | **Rejected for v0.2.0.** The SLA target is organizational policy the runtime cannot know (one shop's "good" is 95%, another's is 99.9%). Hard-coding a threshold would falsify the audit for every shop whose target differs. |
| **Presence-based** — a complete, valid sample is recorded → pass; the rate is surfaced as evidence for the human auditor | **Chosen.** |

This mirrors 9001 exactly: an attestation present → pass, and the auditor reads the statement. A sample present → pass, and the auditor reads the rate (`"49 of 50 …"`). The runtime's job is to **surface reproducible evidence, not to judge organizational thresholds** — per the BWOC-26 split of responsibility ("the dispatcher stamps, downstream tooling decides"). A future `bwoc audit report --below <pct>` flag could threshold the rate; that is not built here.

Concretely, per criterion in declaration order:

- **`expected_evidence_kind = "attestation"`:**
  - attestation present + complete (`statement` + `signer` + `signed_at`) → `status="pass"`, `evidence.kind="attestation"`.
  - present but incomplete → `status="fail"`, `evidence.kind="file"` (`.bwoc/workspace.toml`), remedy names the missing field(s).
  - absent → `status="fail"`, `evidence.kind="file"`, remedy names the attestation block to add.
- **`expected_evidence_kind = "sample"`:**
  - sample present + valid (`summary` + integer `sampled_count` + integer `sampled_of`, `0 ≤ count ≤ of`) → `status="pass"`, `evidence.kind="sample"` with optional `window`.
  - present but incomplete/invalid (missing field, non-integer, `count > of`) → `status="fail"`, `evidence.kind="file"`, remedy names the problem.
  - absent → `status="fail"`, `evidence.kind="file"`, remedy names the sample block to add.

The fail-path `evidence.kind="file"` pointing at `.bwoc/workspace.toml` is the same honest shape BWOC-28 settled on: a missing-evidence finding cannot use `kind="none"` (forbidden with `fail`) nor an empty-value `attestation`/`sample` (forbidden — non-`none` kinds require a non-empty value). The file the operator must edit *is* the reproducible referent.

## How the runtime works (target shape)

```bash
bwoc audit run --plugin audit-iso-20000-1 --json
```

1. Dispatcher spawns `audit.sh` with `BWOC_WORKSPACE`, `BWOC_PLUGIN_DIR`, `BWOC_AUDIT_OPERATION`.
2. `audit.sh` reads `criteria.toml` → per criterion: `id`, `severity`, `expected_evidence_kind`.
3. `audit.sh` reads `.bwoc/workspace.toml`, builds two lookup tables: attestations (keyed by `criterion_id`) and samples (keyed by `criterion_id`).
4. For each criterion in declaration order, route by `expected_evidence_kind` and emit per Decision 3.
5. Dispatcher validates findings against the BWOC-27 schema, emits the canonical envelope. Exit code = `fail` count (BWOC-12).

## Stability

All 8 `criterion_id`s are unchanged from v0.1.0. Adding `expected_evidence_kind` + the per-kind `required` subtables to `criteria.toml` is additive. Plugin version `0.1.0 → 0.2.0` (minor — contract grows, nothing renamed; PLUGINS.en.md §Stability).

## Open follow-ups (out of scope for BWOC-33)

- **Live ticket-system integration (Option C).** A `command`-kind or integration-plugin path that queries the ITSM tool directly, removing the manual transcription step. Anticipated in BWOC-26; needs its own design + integration contract.
- **Threshold reporting.** `bwoc audit report --below <pct>` to flag sample rates under an operator-declared target. The runtime stamps the rate; this would judge it.
- **27001 runtime (BWOC-34).** Reuses both arrays (`attestations` + `samples`); Annex A sampling is SoA-driven, which is its own wrinkle (a dynamic population) — its own note when it starts.
