# 2026-05-27 — A2A P1-serve: HTTP listener + `bwoc a2a` CLI

Gives the merged `bwoc-a2a` protocol core (#71) a real network face: an axum
listener that serves the Agent Card and accepts JSON-RPC `SendMessage`, driven
by two new CLI subcommands. Second slice of the #48 A2A interop epic.

## What changed

- **`bwoc-a2a::serve`** — `serve_blocking(ServeConfig)` runs an axum listener:
  `GET /.well-known/agent-card.json` → the Agent Card; `POST /` → JSON-RPC
  handed to `rpc::dispatch`. A notification (no `id`) returns `204`; malformed
  JSON returns a spec-conformant JSON-RPC error (`-32700` / `-32600`) over HTTP
  200, not a bare 400. Per-request body cap 1 MiB.
- **`bwoc-a2a` binary** (new) — the A2A transport runner. Owns agent resolution
  (workspace registry → manifest + inbox path) plus `card` / `serve`. This is
  where axum/tokio are linked.
- **`bwoc a2a card`/`serve` in the CLI** — a thin **subprocess shim** that
  resolves the `bwoc-a2a` sibling binary and execs it (same resolution as
  `bwoc spawn` → `bwoc-harness`). `card` prints the Agent Card; `serve` runs the
  listener; default bind `127.0.0.1:41241`.
- **Inbox size cap** (`rpc.rs`) — appends are refused once `inbox.jsonl` reaches
  64 MiB, replacing the prior "add a cap later" NOTE now that the inbox is
  network-reachable.

## Decisions

- **Loopback-only by default, no auth in P1.** The listener has no
  authentication yet, so binding defaults to `127.0.0.1`; a non-loopback
  `--bind` is allowed but prints a loud warning. Exposing to a network is an
  explicit operator choice and should wait for the auth phase. (Security
  conservatism over convenience.)
- **A2A serve runs in a sibling binary the CLI execs — not linked.** First cut
  linked `bwoc-a2a` into `bwoc-cli`; the security review caught that this pulls
  axum→tokio→hyper into the CLI's dependency tree, breaking the documented
  dep-quarantine invariant in `bwoc-cli/src/spawn.rs` (`tokio`/`hyper` never
  appear in `bwoc-cli`'s tree). Fix follows the existing precedent: the HTTP
  stack lives in the `bwoc-a2a` binary, and `bwoc a2a` execs it as a subprocess
  exactly like `bwoc spawn` runs the `bwoc-harness` sibling. `bwoc-cli` links no
  HTTP/async deps; `bwoc-core` stays HTTP-free. Both verified via `cargo tree`.
  (A feature-gate alone won't do it — Cargo unifies features across the
  workspace, so any built crate enabling `serve` would re-contaminate the CLI.)
- **`serve_blocking` owns its tokio runtime.** A current-thread runtime is
  created inside `bwoc-a2a`, so the runner stays a plain blocking `main`.
- **Size caps, not per-peer rate limits, for P1.** P1 has no peer identity —
  every inbound message is `from:"a2a"` — so per-peer rate limiting can't be
  attributed yet and is deferred to the auth phase. Total-inbox + per-request
  size caps are the meaningful bounded-growth guards available now.
- **Malformed input → JSON-RPC error, not HTTP 4xx.** The endpoint parses the
  body as raw JSON first so a bad request gets a `-32700`/`-32600` JSON-RPC
  response (Musāvāda: an honest protocol-level answer instead of a transport
  error that an A2A client can't interpret). Also validates `jsonrpc == "2.0"`
  (spec MUST) → `-32600`.
- **JSON-RPC method strings stay PascalCase — do NOT "fix" to `message/send`.**
  Review flagged the `SendMessage`/`GetTask`/… method constants as a possible
  regression to the slash-delimited `message/send`/`tasks/get` form. Verified
  against the current spec (a2a-protocol.org §5.3 *Method Mapping Reference* +
  the SDK source): A2A unified on the proto-derived **PascalCase** method
  strings, which pair with `ROLE_USER`/`TASK_STATE_*`/`agent-card.json`. The
  `message/send` form is the **retired** pre-1.0.0 binding (it pairs with
  `role:"user"`); switching to it would *break* conformance with current peers.
  (Yoniso manasikāra: verified the live spec, not the older design note.)

## Alternatives considered

- *Async CLI (tokio in `bwoc-cli`).* Rejected — would spread the async runtime
  into the CLI for one blocking command; `serve_blocking` keeps the boundary clean.
- *Bind `0.0.0.0` by default.* Rejected — unauthenticated remote inbox writes.
  Loopback default + explicit opt-in is the safe posture until auth lands.
- *Reject malformed bodies with HTTP 400.* Rejected — less useful to an A2A
  client than a JSON-RPC error object.
