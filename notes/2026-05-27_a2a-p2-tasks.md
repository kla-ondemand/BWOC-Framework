# 2026-05-27 — A2A P2: tasks/* ↔ Saṅgha team tasks

Bridges A2A's task methods to a BWOC team's shared task list, so an external
A2A client can see what a Saṅgha team is working on. Third slice of #48 (after
P1-core #71 and P1-serve #72).

## What changed

- **`bwoc-a2a::tasks`** (new) — maps `bwoc_core::team` tasks to A2A: state map
  (`Pending→SUBMITTED`, `InProgress→WORKING`, `Completed→COMPLETED`), `to_a2a_task`
  (team id → A2A `contextId`), and `load_team_tasks` (missing file = empty list).
- **`rpc.rs`** — real handlers: `GetTask` (→ the task or `-32001 TaskNotFound`),
  `ListTasks` (→ `{ "tasks": [...] }`), `CancelTask` (→ `-32002 TaskNotCancelable`).
  `ServeContext` gained an optional `TasksContext { team_id, tasks_path }`.
  `task_id_param` accepts `id`/`taskId`/`name` (binding aliases).
- **`serve`/CLI** — `bwoc a2a serve --team <id>` selects the team whose
  `.bwoc/teams/<id>/tasks.jsonl` is exposed; absent → empty list / not-found.
  Threaded through the `bwoc-a2a` binary and the CLI shim.

## Decisions

- **Lossy-but-honest state mapping.** BWOC's task machine has three states;
  only the three matching A2A states are ever produced. The other A2A states
  (`INPUT_REQUIRED`/`AUTH_REQUIRED`/`FAILED`/`CANCELED`/`REJECTED`) are never
  synthesized — surfacing the model gap rather than inventing states (Musāvāda).
- **`CancelTask` is `-32002 TaskNotCancelable`, not a fake cancel.** BWOC has no
  cancel state and the human lead owns the task lifecycle (`bwoc task`). Faking
  a cancel would lie to the peer; reporting not-cancelable is the truthful answer.
  (When the id genuinely doesn't exist, the more precise `-32001` is returned.)
- **`contextId` = team id.** The shared list is the context an A2A task lives in,
  so the team is the natural A2A `contextId`.
- **Team is opt-in per `serve`.** A2A serves one agent, but tasks live on teams;
  rather than guess the agent's team(s), the operator names the team explicitly.
  No team selected → `tasks/*` behave as an agent with no exposed tasks.

## Alternatives considered

- *Extend `bwoc_core::team::TaskState` with `Canceled`/`Failed`.* Rejected for
  P2 — it reshapes the core task machine for every BWOC user to satisfy one A2A
  verb. Out of scope; `-32002` is the honest interim.
- *Map the Pavāraṇā plan-review gate (`requires_plan` + pending approval) to
  `TASK_STATE_INPUT_REQUIRED`.* Tempting (semantically "waiting on a human") but
  A2A `INPUT_REQUIRED` means waiting on the **client**, not the lead — left as a
  possible later refinement rather than overloading it now.

## Status / deferred

- Done: `tasks` module, three handlers, `--team` plumbing, 25 `bwoc-a2a` tests
  (incl. HTTP-layer `ListTasks`/`GetTask`/`CancelTask` via `tower::oneshot`),
  live `--team` smoke test.
- Deferred: P3 streaming (`SendStreamingMessage`/`SubscribeToTask`, SSE), P4
  outbound client, P5 push. A2A `Task` history/artifacts fields and the
  plan-gate → `INPUT_REQUIRED` refinement are future polish.
